The "rendering-first for live view" step from [[cuda-at-scale-validation]] (live view ≥30 fps, ideally 120+, flat as the map grows — an **architecture-bound**, not compute-bound, goal). Implemented 2026-07-14 via subagent-driven-development from `docs/superpowers/plans/2026-07-13-live-view-fps.md` (12 tasks + 2 whole-branch-review fixes), on `feature/phase6-slam`, commits `ce91718..3f8045b`.

**Component A — off-GUI-thread adaptive mesh:** new `host/src/roomscan/slam/meshprep.py` — `MeshPacket` (plain arrays) + pure `prepare_packet()` + threaded `MeshPrep`. All O(map-size) work (shade / decimate / wall-split / floor-grid extract) runs off the GUI tick; the tick only builds Open3D geometry from a ready packet (`panel._upload_mesh_packet`) at `mesh_upload_hz` (default 3.0). Adaptive decimation is **latched** (fix for a review-found oscillation): decimate to `live_vertex_budget`=150k once a measured upload exceeds `fps_budget_ms`=8.0, then stay decimated; `prepare_packet`'s `source_vertex_count>vertex_budget` guard keeps small maps full-res. **Decimation is display-only** — saved/offline map always from full-res `mapper.mesh()`.

**Component B — pose/mesh transport split:** `wire.pose_message`/`mesh_message` tags (no framing change); `service.serve_client` sends a tiny **pose message every frame** immediately + a **mesh message only when the worker publishes a new mesh** (no more full-trajectory resend); `remote._recv_loop` dispatches by tag and accumulates the trajectory from **pose deltas**. Both `SlamWorker` and `RemoteSlamWorker` keep `latest() -> (mesh, trajectory, FrameStep) | None`.

**Also:** viewport render-fps counter + HUD "VIEW" row (the ≥30/120 metric, distinct from data-fps + `metrics.render_fps`); live trajectory ribbon now **hidden by default** behind a "Trajectory trail" checkbox and throttled when shown (it was rebuilt O(scan-length) on the GUI thread every data frame). Showcase PROCESSING/FINAL path unchanged (`_upload_slam_mesh` untouched).

**ON-RIG BLANK-SURFACE BUG (2026-07-14) — root-caused + fixed.** User reported no live surfaces in SLAM *and* Showcase-preview on `backend=remote`; only the post-process reveal drew surfaces. Root cause: Component B changed the service→client wire format to tagged pose/mesh messages, but the **running GPU container image was built before Task 8** and still sends the OLD untagged combined message (mesh arrays inline, no `"type"` key). The new `RemoteSlamWorker._recv_loop` treated every untagged message as a pose → never entered the mesh branch → `_last_mesh` stayed None forever → panel `mesh` always None → nothing fed to MeshPrep/`_upload_slam_mesh` (pose/traj still flowed, so tracking looked fine). Both live modes share the RemoteSlamWorker, hence both blank. Fixed (c500b0d) by making the client **backward-compatible**: recover an inline mesh in the pose branch (only a legacy service embeds `mesh_v` there; the new POSE message never does) + warn once to rebuild. Lesson: a wire change to a **separately-deployed container** is a protocol-lockstep item — rebuild the image (`tools/slam-container/build.ps1`), and/or keep the client tolerant. Diagnosis method that worked: headless repros of each component in isolation (both delivered meshes) → the failure was the deploy skew, found by diffing old vs new wire format at 5d48b57.

**CONTAINER REBUILT + VERIFIED (2026-07-14).** Ran `tools/slam-container/build.ps1` (new image `b4d3cd278116`, replacing the pre-split `d6495cc4b1ed`) + `stop.ps1`/`start.ps1`. A raw-wire probe against the live `127.0.0.1:5555` service confirmed it now emits the **new tagged format** (58 `pose` + 1 `mesh` message over 58 frames, mesh delivered) — no more legacy untagged combined messages. So the deploy-skew is fully closed AND the pose-per-frame / mesh-only-on-new bandwidth win is now active on the remote path. The client's backward-compat path (c500b0d) stays as belt-and-suspenders for any future stale image. **Rebuild-after-wire-change is now the standing rule for `tools/slam-container/`.**
