# roomscanner-stack — consolidated KiCad project

A single KiCad project that models the **three stacked NUCLEO boards** as one
schematic so you can trace any STM32 pin through the stack to the sensor it
drives:

```
NUCLEO-H563ZI (base)  →  X-NUCLEO-IKS4A1 (middle)  →  X-NUCLEO-53L9A1 (top)
   STM32H563               IMU + environmental          VL53L9CX ToF LiDAR
```

## Validation (real KiCad, not just structural)

Built with `kiutils`, then verified with **KiCad 10.0.3 `kicad-cli`**:

- **ERC: 0 errors, 0 warnings** (`roomscanner-stack-erc.rpt`).
- **Netlist exported** (`roomscanner-stack.net`) — 29 components, 38 nets — and
  cross-checked. The `PB8` (I3C1_SCL) net resolves, in KiCad's own netlister, to:
  ```
  H563   CN5.10
  IKS4A1 CN105.10, U103.1   (NXS0108 host SCL)
  VL53   CN205.10, U205.3   (PI4ULS3V204 host SCL)
  ```
  i.e. one net spanning all three boards — the shared ToF/IMU bus. `PB9` is the
  same for SDA; `IKS_SCL/IKS_SDA` carry all seven IKS4A1 sensors; the power rails
  merge across boards.
- **PDF rendered** (`roomscanner-stack.pdf`).

## Scope — "stack-interconnect model"

Not a full component-level redraw. It captures the shared connectors and the
functional ICs on the stack signal path:

- **Included:** all four ST Zio / Arduino headers (`CN5/6/8/9`) on every board;
  the VL53L9CX + its two PI4ULS3V204 level shifters + 12 MHz oscillator; the
  IKS4A1's NXS0108 I²C shifter, LDK130 LDO, the LSM6DSV16X (HUB1) IMU, **and all
  six secondary sensors** — LIS2MDL (mag), LPS22DF (baro), LIS2DUXS12 (accel),
  STTS22H (temp), LSM6DSO16IS (HUB2 IMU/ISPU), SHT40 (RH/T) — on the internal
  1.8 V I²C bus.
- **Omitted** (never cross the stack): ST-LINK-V3EC, USB, Ethernet PHY, most
  on-board regulators, and the passives/decoupling. The VL53 on-board LDO rails
  (AVDD/DVDD) are abstracted (no-connect with a note).
- **Morpho headers `CN7/CN10`** are drawn as present (2×19 pass-through) but
  their pins are no-connect — the sensors don't use them.

## Jumper / switch configuration

The as-built default jumper positions for all three boards are recorded on each
board section of the schematic (text notes) and tabulated in `jumper-config.md`.
Base-board jumpers (JP2/JP4/JP5/JP6 — power/debug/USB) don't touch the Zio pins,
so they're documentation-only. The one jumper that changes a modeled net is the
53L9A1 **`SW1` = INT**: it's drawn as an actual SPDT component, so the netlist
correctly shows the ToF clock coming from the on-board **Y1 12 MHz** oscillator
while the host `PB5`/`CLK_IN` path dead-ends at the switch (unused in this
default). See `jumper-config.md` for the full table and net effects.

## How connectivity works

One flat A1 sheet, three titled sections (base / middle / top). Nets that are
electrically common across the stack carry a **global label named after the
STM32 pin** (e.g. `PB8`); same-named labels merge, so the netlist proves the
shared bus. Board-internal nets use local labels (`VL53_S_SCL`, `IKS_SCL`, …).

**Reference numbering:** all three boards silk-print the same connector names,
but KiCad needs unique references. The base keeps the real names; the middle
board is offset **+100** (`CN105` = real `CN5`, `U103` = real `U3`) and the top
board **+200** (`CN205` = `CN5`, `U205` = `U5`). A `= CNx` / `= Ux` text sits by
each so the real silk name is visible.

## Datasheets

`datasheets/` holds the PDFs we have locally (copied in so the project is
self-contained and git-trackable — see `datasheets/MANIFEST.md` for provenance).
Each symbol's **Datasheet** field points at the local PDF where we have it
(VL53L9CX, LSM6DSV16X, the three board docs, STM32H563) or at the vendor/ST URL
otherwise (the six secondary sensors, PI4ULS3V204, NXS0108, LDK130). In KiCad,
right-click a symbol → **Show Datasheet**.

## Regenerating

```sh
pip install kiutils
python generate_stack.py     # rewrites .kicad_sch/.kicad_sym/.kicad_pro + re-copies datasheets
```

Edit the canonical connector/IC/sensor data near the top of `generate_stack.py`
and re-run. Every pin assignment's sourcing is documented in `stack-pinmap.md`.
