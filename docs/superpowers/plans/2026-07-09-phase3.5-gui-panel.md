# Phase 3.5: GUI Control Panel — Implementation Plan (SPEC — not yet scheduled)

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Checkbox steps.
> **Status: SPEC'd 2026-07-09 at owner request; execution not yet started.** Owner priority order puts
> Phase 4 (IKS4A1 sensors) next — run this either before Phase 4, in parallel via a second session, or
> after; it touches only the host presentation layer and conflicts with nothing firmware-side.

**Goal:** Replace the bare Open3D window + console HUD with a real control panel: buttons, sliders, a
status pane, and an event log — no more undiscoverable keyboard-only UI.

**Architecture:** Port the viewer to Open3D's `visualization.gui` framework (`gui.Application` +
`SceneWidget` for the cloud + a settings panel). ALL existing plumbing is reused unchanged —
`TransformStage`, `CommandClient` (offer/send split; sends stay off the reader/UI threads),
`Deprojector`, `sources`/pacing, `config.py`. This is a presentation-layer rebuild only; the legacy
`roomscan-view` window stays available behind a `--classic` flag until the panel proves itself (one
release of overlap, then remove).

**Tech Stack:** open3d.visualization.gui / rendering (O3D 0.19 — verify API names against the installed
version first; the gui module's threading rule is: scene/UI mutations only via
`gui.Application.instance.post_to_main_thread`).

## Global Constraints

- All standing conventions (53L9A1 read-only; unsigned commits + trailers; suite green — 97 baseline).
- Threading rules (hard-won, do not violate): serial writes never on the reader thread; UI mutations
  only on the GUI main thread (`post_to_main_thread`); CommandClient sends only from worker threads.
- The console HUD line remains for `--classic` and headless-ish workflows; the panel duplicates it
  visually but the underlying Stats/decoder counters are the same objects — no forked accounting.
- Config keys gain panel-relevant defaults (last-used color mode, panel width) — same CLI > file >
  default precedence via the existing `config.py` (extend, don't fork).

## Panel contents (v1)

| Group | Controls |
|---|---|
| Status | connection state + port; fps (live-computed, both conventions on hover/tooltip); frames / raw / seq gaps / drops / crc / skipped; current usecase + exposure |
| Device | buttons: Ping, Request CALIB, Reinit; usecase selector (AR_RANGE / AR_PRECISION — radio or combobox; AF entries greyed out with "requires binning 4 — unavailable at full res" tooltip); exposure slider 1-30 ms with applied-value echo |
| View | color mode dropdown (depth / reflectance / confidence); point size slider; background toggle; reset-view button |
| Capture | Record start/stop (writes captures/<timestamp>.bin via the existing recorder tee); replay controls when in replay mode (pause/resume, fps slider) |
| Events | scrolling log pane: device EVENTs (decoded), command results, connect/disconnect — the stuff that today interleaves with the console HUD |

## Tasks

### Task 1: GUI skeleton + render loop port
`host/src/roomscan/panel.py` (new; `roomscan-panel` console script + `--panel` flag on roomscan-view
choosing the implementation). gui.Application skeleton: SceneWidget rendering the point cloud from the
existing reader-thread → latest-wins slot (poll via a tick callback or post_to_main_thread from the
reader — pick per O3D idiom, document), settings panel with static layout, close handling (reader
thread + source shutdown — the clean-close path the classic viewer never needed). Replay + live both
render. Verification: supervised runs of both modes; suite green (panel logic that's testable —
state-model classes — gets unit tests; the GUI shell itself is supervised-run verified like the classic
viewer).

### Task 2: Status group + event log
Wire Stats/decoder counters + TransformStage counters into labels updated ≤4 Hz on the main thread;
event log widget consuming the EVENT/ACK prints (route through a small in-process log bus instead of
print() — classic mode keeps printing; panel mode appends to the pane). Tests for the bus.

### Task 3: Device group
Buttons/selector/slider → CommandClient sends from worker threads (reuse CommandKeyState's dispatch +
busy semantics — generalize it out of the key-handler so keys AND buttons share one dispatcher);
usecase selector reflects ACK-applied state (and re-syncs on reconnection via PING+state read...
NOTE: there is no GET_STATE command in the registry — v1 panel tracks last-ACKed state locally and
marks it "assumed" after reconnect; add a GET_STATE command to the registry ONLY if this proves
painful, as its own protocol-change mini-task). Exposure slider debounced (send on release, not drag).

### Task 4: View + Capture groups
Color mode live-switching (TransformStage outputs are already per-frame dicts — switching the colorize
channel needs no pipeline restart; reflectance/confidence need the stage constructed with those outputs:
construct with all three when the DLL is available, document the ~zero marginal cost since one instance
computes them together). Record start/stop via the recorder tee (needs a pump/recorder API tweak to
support mid-stream start/stop — small sources.py change + tests). Replay pause/fps-slider via the
existing pacing hook.

### Task 5: E2E + docs + retro-feed
Live E2E exercising every control on hardware; config persistence round-trip; ROADMAP note (Phase 3.5
✅ when done); classic-mode regression check; keymap banner updated to mention `--panel`; ledger +
retro candidates.

## Risks / honesty

- O3D gui on Windows + this GPU: the framework is heavier than the legacy visualizer; first task must
  include a go/no-go spike (if SceneWidget point-cloud updates can't hit ~30 Hz with 2,268 points —
  they should, trivially — escalate before building the panel on it).
- The gui module's event loop replaces the classic poll loop — pacing/latest-wins semantics need care
  (Task 1 owns proving frame flow before any widgets exist).
- No GET_STATE command exists (see Task 3) — deliberate YAGNI with a named escape hatch.
